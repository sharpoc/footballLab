# Elo 数据源断供应对设计（自算 Elo 取代 eloratings 抓取）

> 状态：设计待确认。本文件只做方案论证，不含实现；实现需另写计划并经用户确认。
> 免责：仅用于研究分析，不构成投注建议。

## 背景与紧迫性

- `www.eloratings.net` 自 2026-06-11 起对非浏览器请求返回 JS 挑战页（"One moment, please..."），抓取拿不到真数据。
- 2026-06-11 06:33 UTC 探测确认：
  - 默认 Python urllib UA → `HTTP 415`。
  - 浏览器 UA → `HTTP 200` 但响应是 JS 挑战页 HTML（不是 TSV）；带 cookie + Referer 重试仍为 `415` / 挑战页。
  - 即"加浏览器请求头"无法绕过，破解需无头浏览器执行 JS，违反"引擎层不联网、不依赖重型组件"约束，且挑战逻辑易变，不可靠。
- 当前生产靠人工恢复的本地 Elo 缓存运行，受 `ELO_CACHE_GRACE_SECONDS = 48h` 宽限期保护。
- **倒计时**：缓存 mtime 为 2026-06-11 01:33 UTC，宽限期 2026-06-13 01:33 UTC（北京时间 6-13 09:33）到期。到期后每轮 refresh 标 `stale_sources=["eloratings"]`，全站 S/A 信号被 `unconfirmed_backup` 压到 B。
- 叠加问题：今晚起小组赛完赛后，官方 Elo 会变、我们的静态缓存不会，模型 Elo 会逐日落后于真实强弱。

## 方案对比

| 方案 | 做法 | 工作量 | 稳定性 | 结论 |
|---|---|---|---|---|
| 1 加浏览器请求头重试 | UA/Referer/cookie | 小 | 无效（已探测否定） | 放弃 |
| 2 人工兜底下载 | 浏览器手动下 TSV 放进 `data/cache` | 零代码 | 稳但需每日人工 | 作为 P0 临时顶班 |
| 3 自算 Elo（增量重放） | 赛前官方快照 + 本届赛果重放推演 | 中 | 自给自足、不依赖网站 | **正解** |

## 方案 3 设计（推荐）

### 现成资产

- `worldcup/elo_replay.py` 已实现 eloratings 公开公式：`k_factor`（按赛事类型）、`goal_index`（净胜球指数）、`update_pair`（零和更新 + 主场修正）、`replay`（按日期顺序重放）。
- 已验证：4.9 万场国际比赛重放，官方 top-10 全部落在 replay top-30（`overlap_hits=10/10`）。
- 即"自算 Elo 引擎"已存在且验证过，本方案是把它从"离线对照工具"接成"生产 Elo 来源"。

### 核心思路：锚定 + 增量

不从 1872 年重放全部历史（误差累积、且要维护全量历史结果），而是：

1. **锚点**：把当前这份人工恢复的、可信的官方 Elo 缓存（244 队 rating）作为基线快照冻结保存，例如 `data/cache/elo_baseline.tsv` + 记录基线日期 `elo_baseline_date`。
2. **增量**：只对"基线日期之后"完赛的本届世界杯比赛（来自已有的 openfootball 完赛比分）跑 `update_pair`，把 delta 叠加到锚点 rating 上。
3. **产出**：生成"赛前最新 Elo"，写入与现有 collector 相同的内部 `EloRating` 结构，pipeline 无感替换。

这样：

- 不依赖 eloratings 网站。
- 只需本届赛果（openfootball 已在抓且稳定），不需维护全量历史结果文件。
- 锚点本身就是官方值，增量场次少（小组赛 72 场），误差可控。

### 取数边界

- 只更新参加 2026 世界杯的队（32 队 + 锚点其余队保持基线值用于对照）。
- K 因子：世界杯决赛圈用对应 `k_factor` 档位（`elo_replay` 已有映射）。
- 中立场：世界杯小组赛除东道主外按中立处理（与现有 pipeline 的 neutral 口径一致；东道主优势已由 pipeline 的 host advantage 单独处理，Elo 增量按中立算，避免双重计入）。
- 完赛判定复用 `parse_openfootball_results`（已兼容 `score1/score2` 与 `score.ft`）。

### 与现有降级规则的衔接

- 自算 Elo 成为主来源后，eloratings 抓取失败**不再触发任何降级**（因为我们本就不依赖它的实时抓取）。
- 保留 eloratings 抓取为"可选对照"：能抓到就和自算值比对、记录偏差到 `data_quality`；抓不到只记 `source_errors`，不标 `stale_sources`。
- `ELO_CACHE_GRACE_SECONDS` 宽限逻辑在自算 Elo 上线后可退役或改为"基线快照新鲜度"语义（实现计划再定）。

### 安全护栏（防"自算跑飞"）

- 上线时用方案 3 产出和当前官方缓存做一次比对：参赛队 rating 偏差应在合理范围（如单队 |Δ| < 一个阈值），偏差超限则报错不静默替换。
- 自算结果同样要过现有 `_validate_world_tsv` 等价的"非空/可解析"校验。

## 实施顺序建议

1. **P0 今天**：落地方案 2 文档化人工兜底步骤（浏览器下 `World.tsv` / `en.teams.tsv` → 覆盖 `data/cache` → mtime 刷新 → 宽限期重新计时），保证 6-13 不断供。属运维步骤，不需改代码。
2. **P1 1-2 天内**：按本设计写方案 3 实现计划（TDD：基线冻结 → 增量重放接线 → 安全护栏比对 → refresh_runner 接入 → 降级规则改写），交付后 Elo 自给自足。

## 非目标

- 不引入无头浏览器 / 不破 JS 挑战。
- 不从 1872 年全量重放（只锚点 + 本届增量）。
- 不改模型参数（`mu_*` / `dc_rho` 等仍按既有"先攒样本"规则）。
- 不在本设计内动 EV/Edge 阈值。
