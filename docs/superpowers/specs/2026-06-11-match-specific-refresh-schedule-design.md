# 单场独立下次更新时间设计

## 目标

把研究台账从“一个全局刷新间隔/下次计划”改为“每场比赛独立计算下次更新时间”。页面上每个有信号的比赛行都能看到自己的“下次更新”，右侧更新规则卡片只说明规则和最近一场待更新计划。

## 背景

现有调度只根据最近一场 `next_kickoff_at` 选择一个全局间隔：

- 常规：24 小时
- 赛前 7 天内：12 小时
- 赛前 3 天内：6 小时
- 赛前 1 天内：2 小时
- 低额度：24 小时

这个策略会在临近比赛时变得不够精确。首发通常在赛前约 60 分钟出现，因此每场需要有自己的临赛锚点。

## 规则

每场比赛按 `kickoff_at_utc` 单独生成 `refresh_plan`：

- 已开赛或已完赛：停止赛前更新。
- 常规：24 小时。
- 赛前 7 天内：12 小时。
- 赛前 3 天内：6 小时。
- 赛前 24 小时内：2 小时。
- 赛前 6 小时内：1 小时。
- 临赛固定锚点：
  - T-3小时30分：临赛第一轮。
  - T-1小时30分：阵容/伤停预热。
  - T-70分钟：首发提前探针。
  - T-55分钟：首发主抓点。
  - T-40分钟：首发确认。
  - T-25分钟：临场最终确认。

低额度时不一刀切丢掉首发窗口：

- `quota_remaining <= 0`：不计划自动刷新，显示额度耗尽。
- `0 < quota_remaining <= 30`：保留 24 小时低频节奏，同时保留 T-90 / T-55 / T-25 三个关键锚点。

如果某个锚点已经越过但上次刷新发生在该锚点之前，则这场比赛立即到期，避免因为 LaunchAgent 唤醒有偏差而漏掉关键点。

## 数据结构

每场比赛输出：

```json
"refresh_plan": {
  "next_update_at": "2026-06-11T17:30:00+00:00",
  "policy_reason": "pre_90m_lineup_warmup",
  "label": "T-1小时30分",
  "description": "阵容/伤停预热",
  "interval_seconds": 3600,
  "should_refresh": false
}
```

全局调度决策仍保留现有字段，并新增 `match_plans`。全局 `next_due_at` 取所有比赛中最早的 `next_update_at`。如果任意一场到期，`should_refresh=true`。

## 展示

表格保留现有“更新”列，继续表示赔率源/分析更新时间。新增“下次更新”列：

```text
下次更新：06/12 01:30
规则：T-1小时30分
```

展开详情中补充：

```text
下次更新：阵容/伤停预热：2026 年 6 月 12 日 星期五 01:30
```

右侧“更新规则”卡片改为说明“按每场比赛独立调度”，并显示最近一场待更新计划。

## 非目标

- 不改模型参数。
- 不调用 The Odds API。
- 不触发 live refresh。
- 不发布线上。
- 不把下注金额、资金、建议类字段加入页面或 API。
